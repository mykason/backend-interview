# Redis 为什么这么快？

## 完全基于内存操作

Redis 最核心的性能基础是基于内存操作。绝大多数请求是纯粹的内存交互，CPU 通常不是 Redis 的瓶颈，瓶颈更多在网络带宽和内存大小。

由于避免了磁盘 I/O 的寻道和读写时间，Redis 的访问速度非常快。

## 高效的数据结构

Redis 没有直接使用 C 语言的字符串或基础结构，而是专门设计了高效的数据结构，例如：

- 动态字符串（SDS）：通过预分配空间避免频繁内存重分配。
- 哈希表：通过渐进式 rehash 避免阻塞。
- Listpack、跳表（SkipList）等结构：在空间和时间复杂度上做了优化。

## 单线程模型

Redis 的网络 I/O 和键值对读写由一个线程完成。

单线程模型的优势：

- 避免多线程上下文切换开销。
- 不需要考虑加锁和死锁问题。
- 内部实现更轻量。

Redis 6.0 引入了多线程，但主要用于网络请求解析和数据返回，核心命令执行仍然是单线程，以保证线程安全和实现简单。

## I/O 多路复用机制

Redis 使用 `epoll`（Linux 下）等非阻塞 I/O 多路复用技术。

一个线程可以同时监听多个 Socket 连接。当 Socket 变得可读或可写时，Redis 才执行对应操作，这让单线程也能高效处理大量并发连接。

# Redis 跳表

## 基本思想：空间换时间

普通单向链表的查询时间复杂度是 `O(N)`。

跳表在底层单链表的基础上增加多级索引层。查询时从最高层开始，通过跳过多个节点快速缩小范围；如果目标值比当前层下一个节点小，就降到下一层继续查找，最终实现类似二分查找的 `O(log N)` 时间复杂度。

## Redis 内部节点实现：ZskiplistNode

### 多层指针（Level）

每个节点包含一个层级数组。插入节点时，Redis 会通过随机算法计算节点层数，大约有 25% 的概率晋升到上一层，最高 64 层。

每一层包含：

- `forward`：指向下一个节点的前进指针。
- `span`：当前节点到下一个节点之间隔了多少个真实节点。

### 跨度（Span）的作用

`span` 用于高效计算元素排名（Rank）。

例如执行 `ZRANK` 命令时，只需要将沿途经过的层级 `span` 累加起来，就能在 `O(log N)` 时间内算出排名。

### 后退指针（Backward）

跳表底层第一层是双向链表，除了前进指针，还有后退指针。

这方便 `ZREVRANGE` 这类逆向范围遍历命令。

### 排序规则

- 先按 Score 从小到大排序。
- 如果 Score 相同，再按 Member 的字典序排列。

## 为什么用跳表，而不是红黑树或 B+ 树？

### 内存占用更灵活

跳表可以通过调整晋升概率控制内存消耗，通常比红黑树的指针开销更灵活。

### 范围查找高效

ZSet 经常需要执行 `ZRANGE` 做范围查询。

红黑树做范围查询时，需要先找到起始点，再进行中序遍历；跳表找到起始点后，可以直接沿着底层第 1 层链表遍历，逻辑更简单高效。

### 实现简单

红黑树插入和删除涉及左旋、右旋和变色等复杂操作。

跳表只需要修改相邻节点指针，在 Redis 单线程模型下实现更清晰，Bug 风险更低。

# Redis Geo 的底层原理

Redis GEO 底层本质上使用的是 ZSet 结构。

它通过 GeoHash 算法，将二维经纬度数据不断二分并交替组合，降维编码成一维的 52 位整数，作为 ZSet 的 Score 存储。

执行附近范围查询时，为避免 GeoHash 网格边界带来的误差，Redis 会计算目标点及其周围 8 个相邻网格的范围，在 ZSet 中做初步筛选，最后再通过精确的球面距离计算得到最终结果。

# Redis 在 CAP 方面的理解

Redis 更偏向 AP 架构，提供的是最终一致性，而不是强一致性。

Redis 通过异步复制和主从切换机制，在容忍网络分区（P）的同时，提供较高的性能和可用性（A）。

但也正因为异步复制，Redis 无法保证强一致性（C）。在主从切换或脑裂场景下，可能存在丢失少量数据的风险，只能保证最终一致性。

如果业务场景对数据一致性要求极高，例如金融核心账本，通常不应将 Redis 作为唯一数据源，可以考虑使用 ZooKeeper、Etcd 等 CP 架构中间件。
