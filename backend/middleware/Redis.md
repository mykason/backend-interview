# Redis 为什么这么快？
- 完全基于内存操作： 这是 Redis 最核心的性能基础。绝大多数请求是纯粹的内存交互，CPU 不是 Redis 的瓶颈，它的瓶颈通常在网络带宽和内存大小。由于避免了磁盘 I/O 的寻道和读写时间，速度自然极快。
- 高效的数据结构： Redis 并没有直接使用 C 语言的字符串或基础结构，而是专门设计了高效的数据结构。比如：动态字符串（SDS）预分配空间避免频繁内存重分配；哈希表渐进式 rehash 避免阻塞；以及压缩列表（Ziplist/Listpack）、跳表（SkipList）等，都在空间和时间复杂度上做到了极致。
- 单线程模型（执行命令的核心流程）： Redis 的网络 I/O 和键值对读写是由一个线程来完成的。单线程最大的好处是避免了多线程上下文切换的昂贵开销，同时完全不需要考虑加锁和死锁的问题，内部实现变得非常轻量级。（注：你可以补充说明 Redis 6.0 引入了多线程，但仅用于网络请求的解析和数据返回，核心的命令执行依然是单线程，保证了线程安全和简单性。）
- I/O 多路复用机制： Redis 采用了 epoll（在 Linux 下）等非阻塞 I/O 多路复用技术。一个线程可以同时监听多个 Socket 连接，当 Socket 变得可读或可写时，才真正去执行相应的操作。这让单线程也能高效处理海量的并发连接。

# Redis 跳表
- 基本思想（空间换时间）： 普通的单向链表查询时间复杂度是 $O(N)$。跳表在底层单链表的基础上，增加了多级“索引层”。查询时从最高层开始找，通过跳过多个节点快速缩小范围，如果比当前层下一个节点小，就“降级”到下一层继续找，最终实现了类似二分查找的 $O(\log N)$ 时间复杂度。
- Redis 内部节点的具体实现（ZskiplistNode）： * 多层指针（Level）： 每个节点包含一个层级数组。插入节点时，Redis 会通过随机算法计算出这个节点的层数（大概是 25% 的概率晋升到上一层，最高 64 层）。每一层不仅包含指向下一个节点的 forward 前进指针，还包含一个 span（跨度）。
-- 跨度的作用（核心亮点）： 记录当前节点到下一个节点之间隔了多少个真实节点。这是为了极其高效地计算元素的 Rank（排名）。比如 ZRANK 命令，只需要把沿途经过的层级 span 累加起来，就能在 $O(\log N)$ 时间内算出排名。
-- 后退指针（Backward）： 底层的第一层是一个双向链表，除了前进指针，还有后退指针。这方便了 ZREVRANGE 这种逆向范围遍历的命令。
-- 排序规则： 节点先按 Score（分数）从小到大排序；如果 Score 相同，再按 Member 成员数据的字典序排列。
- 为什么用跳表不用红黑树或 B+树
-- 内存占用： 跳表可以通过调整晋升概率控制内存消耗，通常比红黑树的指针开销更灵活。
-- 范围查找极其高效： ZSet 经常需要执行 ZRANGE 找区间。红黑树找区间需要先找到起始点再中序遍历，而跳表找到起始点后，直接沿着底层的第 1 层级链表遍历即可，逻辑极其简单高效。
-- 实现简单： 红黑树的插入和删除涉及复杂的左旋、右旋和变色，而跳表只需要修改相邻节点的指针，在单线程下实现起来更清晰，Bug 率更低。

# Redis Geo 的底层原理
- Redis GEO 的底层本质上使用的是 ZSet 结构。它巧妙地利用 GeoHash 算法，将二维的经纬度数据通过不断二分和交替组合，降维编码成一维的 52 位整数，作为 ZSet 的 Score 存起来。在执行附近范围查询时，为了避免 GeoHash 网格边界造成的误差，Redis 会计算出目标点及其周围 8 个相邻网格的范围，在 ZSet 中进行初步筛选，最后再经过精确的球面距离计算，得出最终结果。”

# Redis在CAP方面的理解
- Redis 选择了 AP 架构，提供的是“最终一致性”而非“强一致性
- 它通过异步复制和主从切换机制，在容忍网络分区（P）的同时，提供了极高的性能和可用性（A）。但也正因为异步复制，它无法保证强一致性（C），在主从切换或‘脑裂’场景下存在丢失小部分数据的风险，只能保证最终一致性。如果我们的业务场景对数据一致性要求极度严苛（比如金融核心账本），我们通常不会把 Redis 作为唯一的数据源，或者会转向使用 Zookeeper/Etcd 这种 CP 架构的中间件
